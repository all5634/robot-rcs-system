# 机器人管理服务系统开发文档
## 一、引言
### 1.1 概述
本文档整合机器人管理服务的需求规格与设计实现要求，明确系统的功能需求、接口规范、架构设计、模块实现及部署要求，为系统的开发、测试、部署及验收提供完整且统一的依据。本系统为轻量级、高可用的机器人管理服务（RCMS），作为机器人管理核心后端服务，实现机器人接入管理、运行状态采集、在线状态判断及控制指令下发等功能，同时为前端监控看板提供统一数据支持，且系统不负责具体数据存储，通过接口对接外部数据服务实现解耦设计。

### 1.2 项目背景
随着自动化设备的普及，亟需一个统一的后端系统对机器人进行集中化管理与监控。本项目旨在开发一款满足实时监控、远程控制需求的机器人管理服务，实现机器人全生命周期的状态管理与指令交互，为前端监控系统提供稳定、标准的数据接口。

## 二、总体描述
### 2.1 系统目标
1. 实现机器人的注册接入与基础信息统一管理，为每个机器人分配唯一身份标识。
2. 完成机器人运行状态、心跳数据的采集与处理，精准判断机器人在线/离线状态。
3. 支持向前端监控看板提供机器人列表、详情等查询接口，实现状态实时可视化。
4. 实现对机器人的远程控制指令下发，确保指令正常转发并获取执行反馈。
5. 与外部数据服务对接完成数据持久化，自身聚焦业务逻辑处理，实现系统解耦。
6. 保证系统高可用、高并发支持能力，满足多机器人同时在线的实时监控需求。

### 2.2 技术栈
- 开发语言：Java 17
- 核心框架：Spring Boot
- 构建工具：Maven
- 版本管理：Git（托管于GitHub平台）
- 通信协议：HTTP / TCP/IP
- 接口规范：RESTful API
- 数据格式：JSON

### 2.3 系统整体架构
本系统采用**分层架构+无状态设计**，整体由**机器人设备、后端服务、前端监控看板、外部数据存储服务**四部分组成，各部分职责与交互关系如下：
1. 机器人设备：作为数据采集与指令执行终端，通过HTTP/TCP/IP协议向后端服务上报注册信息、心跳数据、状态数据，接收并执行后端下发的控制指令。
2. 后端服务：系统核心，采用Controller层、Service层、Client层三层架构，负责处理所有业务逻辑、维护机器人内存状态、分发控制指令，通过接口与外部数据服务交互。
3. 前端监控看板：通过RESTful API调用后端服务，获取机器人状态数据并实现实时可视化展示，同时提供人工下发控制指令的操作入口。
4. 外部数据存储服务：负责系统所有数据的持久化存储，后端服务通过Client层调用其接口完成数据的保存与查询，系统自身不直接实现数据库操作。

系统核心运行数据通过内存缓存维护，提升响应速度且支持横向扩展，无状态设计保障系统的高可用与扩展性。

## 三、功能需求与设计
### 3.1 核心功能模块
系统包含四大核心功能模块，各模块职责与实现逻辑明确，解耦良好，支撑系统全部核心业务能力：

| 模块名称       | 核心职责                                                                 | 关键实现逻辑                                                                 |
|----------------|--------------------------------------------------------------------------|------------------------------------------------------------------------------|
| 机器人管理模块 | 机器人设备注册、基础信息维护、唯一身份标识分配                           | 注册请求参数校验，已存在机器人直接返回成功，新机器人调用外部数据服务存储信息并生成robotId |
| 心跳管理模块   | 接收机器人心跳上报、维护最后心跳时间、自动判断并标记机器人在线/离线状态   | 心跳上报时更新内存中最后心跳时间；定时任务检测超时，超期则标记为离线并同步状态 |
| 状态监控模块   | 接收机器人运行状态上报、处理并维护机器人实时状态、为前端提供状态查询支持   | 解析状态上报数据（工作状态、电量、位置），更新内存状态并同步至外部数据服务     |
| 控制指令模块   | 接收前端控制请求、下发控制指令至机器人、记录指令信息并获取执行结果反馈   | 控制指令加入队列，异步线程转发至机器人；记录指令ID与状态，支持指令执行结果查询 |

### 3.2 核心业务流程
#### 3.2.1 机器人注册流程
1. 机器人向后端服务发送注册POST请求，携带robotCode、model、capabilities参数；
2. 后端Controller层接收请求并转发至Service层，Service层进行参数校验；
3. 若机器人已存在（通过robotCode判断），直接返回注册成功；
4. 若为新机器人，Service层调用Client层接口，将基础信息存储至外部数据服务；
5. 系统为机器人生成唯一robotId，作为后续交互的唯一身份标识，并返回给机器人。

#### 3.2.2 心跳管理流程
1. 机器人周期性向后端服务发送心跳上报POST请求，携带robotId、timestamp参数；
2. 后端Service层接收请求，更新该robotId对应的**最后心跳时间**（内存缓存+外部数据服务同步）；
3. 系统启动**定时任务**，持续检测所有机器人的最后心跳时间；
4. 若某机器人超过设定时间未上报心跳，系统自动将其在线状态标记为离线，并更新内存与外部数据服务中的状态；
5. 机器人恢复心跳上报后，系统即时将其状态切换为在线。

#### 3.2.3 状态上报流程
1. 机器人定期向后端服务发送状态上报POST请求，携带robotId、status（工作状态）、battery（电量）、position（位置{x,y}）参数；
2. 后端Service层接收并解析状态数据，实时更新该机器人的内存缓存状态；
3. 同时调用Client层接口，将状态数据及上报时间存储至外部数据服务，形成状态记录；
4. 前端可通过查询接口获取该机器人的实时状态数据。

#### 3.2.4 远程控制流程
1. 操作人员通过前端监控看板发送控制请求，指定robotId并携带commandType（指令类型）、params（指令参数）；
2. 后端Controller层接收请求，Service层将控制指令加入执行队列，并生成唯一commandId；
3. 系统通过异步线程将指令转发至对应机器人，同时将指令信息（commandId、robotId、commandType、params、创建时间）存储至外部数据服务；
4. 机器人接收并执行控制指令，将执行结果返回至后端服务；
5. 后端服务更新该commandId对应的指令状态，前端可通过查询接口获取指令执行结果。

#### 3.2.5 前端查询流程
1. 前端监控看板通过GET请求向后端服务发起机器人列表/详情查询，携带指定robotId（详情查询）或无参数（列表查询）；
2. 后端Service层从内存缓存中读取机器人实时状态数据（在线状态、最后心跳、电量、位置等），若需历史数据则调用Client层从外部数据服务查询；
3. 后端将整理后的结构化数据通过JSON格式返回至前端；
4. 前端对数据进行解析并实现可视化展示。

### 3.3 概念数据模型
系统涉及**机器人、状态记录、控制指令**三大核心实体，实体间均为**一对多**关系（一个机器人对应多条状态记录/多条控制指令），各实体核心属性如下：

| 实体名称       | 核心属性                                                                 |
|----------------|--------------------------------------------------------------------------|
| 机器人（robot info） | robotId（主键）、robotCode、model、capabilities、online（在线状态）、lastHeartbeat（最后心跳时间） |
| 状态记录（robot_status） | id（主键）、robotId（外键）、status（工作状态）、battery（电量）、position（x,y）、timestamp（上报时间）、update_time（更新时间） |
| 控制指令（robot_command） | id（主键/commandId）、robotId（外键）、commandType（指令类型）、params（指令参数）、status（指令执行状态）、create_time（创建时间） |

## 四、详细设计
### 4.1 后端分层架构设计
后端服务采用**三层架构**设计，各层职责单一、解耦良好，便于系统扩展与维护，层间调用关系为**Controller层 → Service层 → Client层**，无反向调用。

#### 4.1.1 Controller层
- 核心职责：接收外部（机器人、前端）的HTTP请求，进行请求参数的初步校验，暴露系统所有RESTful API接口，将请求转发至Service层处理，并将Service层的处理结果封装为统一格式返回。
- 无业务逻辑处理，仅负责请求与响应的转发和封装。

#### 4.1.2 Service层
- 核心职责：系统核心业务逻辑处理层，实现机器人注册、心跳维护、状态更新、指令下发、在线状态判断等所有核心业务功能；维护机器人的内存缓存状态；协调Client层完成数据的持久化操作。
- 核心方法：registerRobot()、updateHeartbeat()、updateStatus()、sendCommand()等。

#### 4.1.3 Client层
- 核心职责：作为系统与外部数据服务的交互桥梁，封装与外部数据服务的对接接口，提供数据持久化的调用方法，Service层通过调用该层方法完成数据的保存与查询。
- 核心方法：saveRobot()、saveStatus()、saveCommand()等，方法与外部数据服务的接口一一对应。

### 4.2 UML核心类关系
系统核心类围绕三层架构设计，核心类及方法、调用关系如下：
1. **RobotController**：Controller层核心类，暴露所有系统API接口，接收外部请求并转发至RobotService。
2. **RobotService**：Service层核心类，实现所有核心业务逻辑，包含registerRobot()、updateHeartbeat()、updateStatus()、sendCommand()等核心方法，调用DataClient类的方法完成数据交互。
3. **DataClient**：Client层核心类，封装与外部数据服务的对接，包含saveRobot()、saveStatus()、saveCommand()等数据持久化方法，为Service层提供数据操作支持。

### 4.3 数据库逻辑设计
系统**不直接实现数据库的创建与操作**，所有数据由外部数据服务进行存储与管理，仅在逻辑上设计以下三张核心表结构，作为与外部数据服务对接的依据：

| 表名           | 表作用                                                                 | 核心字段                                                                 |
|----------------|------------------------------------------------------------------------|--------------------------------------------------------------------------|
| robot_info     | 存储机器人基础信息与实时在线状态                                       | robotId(主键)、robotCode、model、capabilities、online、lastHeartbeat     |
| robot_status   | 存储机器人运行状态历史记录，支持状态追溯                               | id(主键)、robotId(外键)、status、battery、position_x、position_y、timestamp、update_time |
| robot_command  | 存储所有下发给机器人的控制指令及执行状态，支持指令结果查询             | id(主键/commandId)、robotId(外键)、commandType、params、status、create_time |

## 五、接口设计
### 5.1 接口设计原则
1. 遵循**RESTful**设计规范，通过统一的资源路径和标准HTTP方法（POST/GET）实现不同操作；
2. 数据传输格式统一为**JSON**，保证接口的一致性与可扩展性；
3. 所有接口返回数据包含**状态码、提示信息、业务数据**三部分，保证调用结果的统一性和可读性；
4. 接口路径采用版本化设计（/api/v1/），便于后续接口升级与兼容。

### 5.2 核心接口详情
系统所有核心接口均为HTTP接口，涵盖机器人注册、心跳上报、状态上报、信息查询、控制指令下发/查询等所有业务场景，详细信息如下：

| 功能           | 请求方法 | 接口路径                                  | 请求参数（JSON格式）| 响应data字段                          | 接口说明                     |
|----------------|----------|-------------------------------------------|-------------------------------------------------------|---------------------------------------|------------------------------|
| 机器人注册     | POST     | /api/v1/robots                            | {robotCode, model, capabilities}                       | {robotId}                             | 机器人首次接入时完成注册，生成唯一robotId |
| 心跳上报       | POST     | /api/v1/robots/{robotId}/heartbeat        | {timestamp}                                           | 无                                    | 机器人周期性上报，更新最后心跳时间 |
| 状态上报       | POST     | /api/v1/robots/{robotId}/status           | {status, battery, position:{x,y}}                     | 无                                    | 机器人上报实时运行状态       |
| 查询机器人列表 | GET      | /api/v1/robots                            | 无                                                    | [{robotId, robotCode, model, online}] | 前端获取所有机器人基础信息与在线状态 |
| 查询机器人详情 | GET      | /api/v1/robots/{robotId}                  | 无                                                    | {robotId, robotCode, model, online, lastHeartbeat, battery, position} | 前端获取单个机器人全量实时信息 |
| 发送控制指令   | POST     | /api/v1/robots/{robotId}/commands         | {commandType, params}                                 | {commandId}                           | 前端下发控制指令，生成唯一commandId |
| 查询指令状态   | GET      | /api/v1/robots/{robotId}/commands/{commandId} | 无                                                    | {commandId, status}                   | 前端查询指定指令的执行状态   |

## 六、非功能需求
### 6.1 性能要求
1. 支持**多台机器人同时在线**，满足实际场景中的规模化接入需求；
2. 高并发场景下保持**较低的响应延迟**，确保心跳、状态上报的实时性，以及前端查询、指令下发的快速响应；
3. 内存缓存与外部数据服务的同步效率满足实时监控要求，状态更新无明显延迟。

### 6.2 可用性要求
1. 系统具备较高的稳定性，能够**长时间连续运行**，满足7×24小时的监控需求；
2. 具备基本的**异常处理机制**，接口调用失败、服务异常时返回明确的状态码和提示信息，便于问题定位；
3. 具备**故障恢复能力**，轻微异常后可自动恢复运行，不影响整体业务；
4. 心跳判断机制精准，能够实时、准确反映机器人的实际在线/离线状态，无误判、漏判。

### 6.3 日志要求
1. 采用**分级日志策略**，对关键操作记录INFO级日志，对潜在问题记录WARN级日志，对错误异常记录ERROR级日志；
2. 日志包含关键信息：接口调用时间、请求参数、robotId/commandId、处理结果、异常堆栈信息等；
3. 日志可落地存储，支持后期的问题排查、业务分析与数据追溯。

### 6.4 部署要求
1. 支持在**Linux服务器环境**下部署运行；
2. 支持**容器化技术**部署，提升部署效率、环境一致性与可移植性；
3. 后端服务以**Spring Boot应用**形式运行，启动方式简单，支持配置文件灵活修改（如定时任务超时时间、外部数据服务接口地址等）。

### 6.5 版本管理要求
1. 项目使用**Git**进行版本控制，代码托管于**GitHub**平台；
2. 遵循规范的**分支管理策略**（如主分支main、开发分支dev、功能分支feature/*、修复分支bugfix/*），保证开发流程的有序进行；
3. 提交代码时附带清晰的提交信息，便于版本追溯与团队协作。

## 七、系统部署
### 7.1 部署架构
系统采用分布式部署方式，各组件独立部署、通过网络通信，部署架构如下：
```
机器人设备（多终端）→ 网络（HTTP/TCP/IP）→ 后端服务（Spring Boot应用，Linux/容器）→ 外部数据存储服务 → 前端监控看板（浏览器访问）
```
### 7.2 部署步骤
1. 后端服务：将Spring Boot项目通过Maven打包为JAR包，在Linux服务器/容器中启动，配置外部数据服务的接口地址、端口、日志路径等参数；
2. 外部数据服务：部署并启动外部数据存储服务，确保其接口正常提供，且与后端服务的网络互通；
3. 机器人设备：配置后端服务的接口地址与端口，确保机器人能够正常访问后端服务；
4. 前端监控看板：部署前端项目，配置后端服务的API接口地址，通过浏览器即可访问使用。

### 7.3 网络要求
所有组件间网络互通，支持HTTP/TCP/IP协议，确保机器人上报数据、后端转发指令、前端查询数据的网络通畅，无延迟或丢包。

## 八、验收标准
### 8.1 功能验收
1. 系统完整实现**机器人注册、心跳维护、状态上报、信息查询、远程控制**所有核心功能，无功能缺失；
2. 所有接口能够正常调用，请求参数校验有效，返回结果符合预期，无逻辑错误；
3. 心跳机制正常工作，超期未上报心跳的机器人能够被自动标记为离线，恢复上报后即时切换为在线；
4. 控制指令能够正常下发至机器人，机器人执行后结果能够正常反馈，指令状态更新及时。

### 8.2 性能验收
1. 系统在**高并发场景**下（多机器人同时上报/前端多次查询）能够稳定运行，无崩溃、无响应超时；
2. 所有接口的**响应时间**满足实时监控要求，无明显延迟；
3. 内存缓存更新及时，外部数据服务同步效率符合预期。

### 8.3 接口验收
1. 所有接口严格遵循**RESTful设计规范**，路径、请求方法、数据格式符合设计要求；
2. 接口返回格式统一，包含状态码、提示信息、业务数据三部分，前端能够顺利解析与对接；
3. 接口具备良好的**可扩展性**，新增功能时可通过版本化或新增子路径实现，不影响原有接口使用。

### 8.4 稳定性验收
1. 系统经过**长时间连续运行测试**（如72小时），运行稳定，无内存泄漏、无服务崩溃等问题；
2. 日志记录完整、分级清晰，能够支持问题快速定位；
3. 异常场景下（如网络中断、外部数据服务临时不可用），系统能够做出合理处理，返回明确错误信息，恢复后可正常运行，无数据丢失。

### 8.5 部署验收
1. 系统能够在**Linux服务器/容器环境**下成功部署并启动，启动过程无报错；
2. 所有配置文件可灵活修改，配置参数生效正常；
3. 各组件间网络通信正常，机器人、后端服务、外部数据服务、前端看板能够正常交互。
